# Схема базы данных проекта VentSearch

Данный документ описывает структуру данных, индексы и техническое обоснование архитектуры БД для обеспечения быстрого поиска (менее 2 сек) по каталогу из 20 000+ позиций.

## 1. ER-модель (Сущность-Связь)

Архитектура ориентирована на высокую скорость чтения. Основная сущность — `Product` (Вентилятор).

### Таблица `products` (Товары)
| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `id` | UUID (PK) | Уникальный идентификатор устройства |
| `model` | VARCHAR | Название модели для текстового поиска |
| `type` | VARCHAR | Категория (осевой, радиальный и т.д.) |
| `size` | VARCHAR | Типоразмер или серия |
| `price` | DECIMAL | Стоимость единицы |
| `power` | DECIMAL | Потребляемая мощность (кВт) |
| `noise_level` | DECIMAL | Уровень шума (дБ) |
| `diameter` | DECIMAL | Присоединительный диаметр (мм) |
| `airflow` | JSONB | Диапазоны расхода воздуха (min/max) |
| `pressure` | JSONB | Диапазоны давления (min/max) |

## 2. Проектирование индексов

Для поддержки алгоритмов, реализованных в `CatalogIndex`, требуются следующие индексы:

1. **B-Tree на числовые поля** (`price`, `power`, `noise_level`, `diameter`):
   - *Зачем:* Обеспечивает эффективную выборку диапазонов, которая затем обрабатывается классом `SortedRangeIndex`.
2. **Hash или B-Tree на категориальные поля** (`type`, `size`):
   - *Зачем:* Позволяет методу `CatalogIndex.build` мгновенно извлекать уникальные значения для инициализации фильтров Блума.
3. **GIN индекс на поля `airflow` и `pressure` (JSONB)**:
   - *Зачем:* Ускоряет проверку вхождения запрашиваемых значений в рабочие диапазоны вентилятора.

## 3. Техническое обоснование

### Двухэтапная фильтрация
Система использует Bloom Filter для предварительной проверки `type` и `size`. Это позволяет мгновенно отсечь категории, которых точно нет в базе, не нагружая основной индекс БД.

### Логарифмическая сложность
Применение `SortedRangeIndex` и бинарного поиска (`bisect`) гарантирует, что поиск по числовым параметрам выполняется за $O(\log N)$. Это критично для масштабирования системы до 20 000+ записей.

### Денормализация
Поля `type` и `size` хранятся в основной таблице (без выноса в справочники), чтобы минимизировать количество `JOIN` при построении поискового индекса в оперативной памяти.
